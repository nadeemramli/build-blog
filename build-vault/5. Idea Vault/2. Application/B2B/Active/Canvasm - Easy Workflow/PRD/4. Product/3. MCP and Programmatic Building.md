# MCP and Programmatic Building — Design

Design doc for the **Metrimap MCP + API** capability. This is the shaped output of the design spike (Linear **CVS-97**), so the build issues **CVS-98 → CVS-104** can proceed. Project: *Programmatic building — Metrimap MCP + API*.

## Why

Users shouldn't hand-build every driver/metric tree. Let their **own agents** (Claude Cowork/Code, Codex) — using **Composio** and other MCPs to pull data from GA4, Stripe, warehouses, etc. — **build and populate trees in Metrimap**. Metrimap becomes the **visualization / abstraction layer**; the agent does the gathering.

## Two ingestion paths — don't confuse them

| Path | Who fetches | Mechanism | Issue |
|---|---|---|---|
| **Metrimap pulls** | Metrimap | OAuth to GA4/Stripe; scheduled sync | CVS-90 (Connected accounts) |
| **Agent pushes** (this) | The user's agent | Agent pulls elsewhere, **pushes** structured tree + data via our MCP | CVS-97–104 |

Both feed the same metric cards; this doc is the **push** side.

## Architecture

```mermaid
flowchart LR
  A["User's agent<br/>Claude / Codex + Composio"] -->|MCP tools| B["Metrimap MCP server<br/>mcp.canvasm.app (HTTP/SSE)"]
  B -->|auth context (user)| C["API / service layer<br/>RLS-scoped"]
  C --> D["Supabase (RLS)"]
  A -->|push_values / upload_csv| E["Staging store<br/>import tables / CSV, TTL"]
  E -->|map + materialize| F["metric_values"]
  D --> G["Canvas visualization"]
  F --> G
```

## Decisions (what the spike settles)

### 1. Auth model — **OAuth 2.1 primary + personal API key fallback**
- **OAuth ("Connect" / sign-in):** adding the server in Claude (claude.ai connector or `claude mcp add`) triggers a sign-in/authorize flow. Best UX — "sign in → connected."
- **Acting as the user without the service-role key:** the OAuth flow yields a per-connection **Clerk** token for that user; the server calls Supabase through the existing **Clerk→Supabase authenticated client** (`createClerkSupabaseClient(getToken)`), so **every DB call runs under that user's RLS**. The service-role key is never used.
- **API key fallback (CVS-89)** for headless/CLI (Codex): the key resolves to a user; the server exchanges it for a short-lived scoped token and calls under RLS.

### 2. Transport & hosting — LOCKED
- **Transport: Streamable HTTP** (SSE is deprecated; this future-proofs it and keeps a serverless option open later).
- **Hosting: Node container on a Hetzner VPS** via Docker Compose, **shared with background workers** — see the *Shared infra & workers (Hetzner VPS)* project and vault note **7. Shared Infra (Hetzner VPS)**. Not Vercel serverless (SSE/duration/stateless limits). Deploy = `npm i` → set host env (incl. `SUPABASE_JWT_SECRET`) → `docker compose up`.
- Reachable at `mcp.canvasm.app` by `claude mcp add --transport http …` and claude.ai connectors.

### 3. Tool surface (v1)
- **Build:** `list_canvases`, `create_canvas`, `create_metric`, `create_driver_node`, `create_action`, `create_relationship` (typed: deterministic / probabilistic / causal / compositional), `get_tree`, `update_node`, `delete_node`.
- **Data:** `push_values` (series), `upload_csv`, `map_and_materialize`.
- Typed schemas + **strong descriptions** (agents lean on them); return ids for chaining; `get_tree` gives context so agents **extend** rather than duplicate. Apply **auto-layout** so pushed trees render sensibly.

### 4. Ingest / staging model
- `push_values` / `upload_csv` land in **staging** (`import_batches` + `import_rows`, or a stored CSV blob) scoped per user/canvas, with a **TTL** (auto-expire).
- A **mapping** step (agent-specified in the push, and/or an in-app mapping UI) maps columns/series → metric cards.
- **Materialize** into `metric_values` (idempotent upsert); return an ingest report (rows in / mapped / errors). Metrimap is the abstraction — the canvas visualizes the materialized values; raw imports are transient.

### 5. Data-model additions
- `api_keys` (hashed, per-user, scopes, last_used) — CVS-89.
- OAuth client/token storage (or reuse Clerk's).
- `import_batches` / `import_rows` (staging, TTL).
- `mcp_audit_log` (tool calls: who / what / when).
- All RLS-scoped.

### 6. Security
- RLS everywhere; per-connection **scopes** (read vs write); **rate limits** + payload caps (esp. CSV); **audit log**; safe failure (never partial-corrupt a tree). Load/security review before public exposure. (CVS-104)

## Phased plan (Linear)

`CVS-97` spike (this) → **`CVS-98`** API/service layer → **`CVS-99A/B/C`** auth foundation → **`CVS-100`** MCP server scaffold → **`CVS-101`** build tools · **`CVS-102`** data ingest · **`CVS-103`** connect UX + docs · **`CVS-104`** security. Critical path: spike → (API + auth) → server → the rest.

## CVS-99 Full Requirement - Auth For MCP/API

Working title:

**Authenticate programmatic MCP/API calls as a real Metrimap user under RLS.**

### Problem

The current phrase "auth (OAuth + key)" is too vague. The actual job is not simply to add a login screen or validate a token. `CVS-99` must create the authentication boundary that lets external agents and API clients call Metrimap without bypassing workspace permissions or Supabase RLS.

Without this, `CVS-100` through `CVS-104` can accidentally build tools that work technically but are unsafe: they may read/write the wrong workspace, require a service-role bypass, lack scopes, or be impossible to audit.

### Outcome

Programmatic requests can authenticate through either:

1. OAuth / Clerk-backed user auth for connected MCP clients.
2. Personal API key fallback for headless tools such as Codex, CLI agents, scripts, or customer-owned automation.

Both paths must resolve to the same internal request context:

- user id,
- Clerk user id,
- workspace/org id when supplied,
- scopes,
- auth method,
- key/connection id when applicable,
- request id for audit,
- and an RLS-scoped Supabase client.

### Dependencies

- `CVS-89`: personal API keys exist or the key-storage contract is available.
- `CVS-98`: API/service layer has reusable service functions that can receive an authenticated request context.
- Current Clerk -> Supabase authenticated client pattern remains the preferred way to satisfy RLS.

### Scope

#### 1. Lock the auth decision

Confirm and document the v1 auth model:

- OAuth primary path: Clerk-backed user authentication for MCP clients where possible.
- API key fallback: personal scoped key for headless/client-owned automation.
- No product read/write should use Supabase service role as the normal path.
- If full OAuth 2.1 remote MCP compliance is too large for this issue, document the smaller v1 and leave full connector UX/metadata wiring to `CVS-103`.

Decision to write down:

- Is Metrimap acting as its own OAuth authorization server, or delegating to Clerk/OIDC?
- How does an API-key request become an RLS-valid request context?
- Which claims are required for workspace-scoped access?

#### 2. Add auth middleware / request resolver

Create a shared programmatic-auth resolver used by MCP/API routes.

It should accept:

- `Authorization: Bearer <oauth_or_clerk_token>`
- `Authorization: Bearer <personal_api_key>`
- optional workspace/canvas target in request body, query, or header.

It should return:

- normalized auth context,
- allowed scopes,
- resolved user,
- resolved workspace if provided,
- safe error if invalid.

#### 3. Validate OAuth / Clerk-backed access

For OAuth/user-token requests:

- validate the token against Clerk/JWKS or the existing Clerk auth mechanism,
- resolve the Metrimap user,
- resolve accessible workspace/org,
- create a Supabase client that runs under that user's RLS,
- reject expired, invalid, or anonymous tokens.

#### 4. Validate personal API keys

For API-key requests:

- hash the presented key before lookup,
- reject missing, revoked, expired, or malformed keys,
- check key scopes,
- check workspace restrictions if the key is workspace-scoped,
- update `last_used_at` / usage metadata safely,
- never log raw keys,
- return the same normalized auth context as the OAuth path.

Minimum key scopes:

- `mcp:read`
- `mcp:write`
- `canvas:read`
- `canvas:write`
- `metrics:write`
- `evidence:write`

Scope names can change, but the issue must lock a v1 set.

#### 5. Enforce workspace and canvas access

Before a programmatic call can read/write a canvas:

- confirm the authenticated user can access the target workspace,
- confirm the user can access the target canvas/project,
- reject cross-workspace writes,
- reject ambiguous workspace selection when the user belongs to multiple workspaces and no target is provided.

This should use existing permission helpers where possible.

#### 6. Create RLS-scoped Supabase client factory

Add a single helper for programmatic calls:

```ts
createProgrammaticSupabaseClient(authContext)
```

It must:

- run as the resolved user,
- preserve RLS behavior,
- support both OAuth and API-key auth paths,
- expose enough request metadata for service/audit code,
- avoid service-role product reads/writes except for narrow internal key lookup or metadata operations if absolutely required.

If a service-role lookup is unavoidable for key validation, that must be tightly isolated from product data access.

#### 7. Add audit hooks

Add or prepare an audit event payload for future `mcp_audit_log`:

- request id,
- user id,
- workspace id,
- auth method,
- scope used,
- tool/route name,
- target canvas/metric/evidence id when applicable,
- success/failure,
- error class,
- timestamp.

This issue does not need to build the full audit UI.

#### 8. Add safe errors and logging

Return predictable errors:

- `401` unauthenticated,
- `403` insufficient scope or workspace access,
- `404` target not found or inaccessible,
- `429` rate limited if key-level limits already exist,
- `500` unexpected internal error without leaking secrets.

Logs must never include raw API keys, access tokens, refresh tokens, Supabase service keys, database URLs, or customer payloads.

### Non-goals

- Do not build MCP tools themselves. That belongs to `CVS-101` and `CVS-102`.
- Do not build the MCP server scaffold. That belongs to `CVS-100`.
- Do not build native source OAuth connectors like GA4/Stripe. That belongs to `CVS-90` and connector work.
- Do not build the full connect/settings UX. That belongs to `CVS-103`.
- Do not implement broad workflow execution.
- Do not bypass RLS with service-role product reads/writes.

### Acceptance Criteria

- OAuth/Clerk-authenticated programmatic request resolves to a user and can call a test protected route under RLS.
- Personal API key request resolves to the same normalized auth context and can call the same protected route if it has scope.
- Request without auth is rejected.
- Request with invalid/revoked/expired key is rejected.
- Request with missing scope is rejected.
- Request targeting an inaccessible workspace/canvas is rejected.
- Multi-workspace user must explicitly select a workspace when the target is ambiguous.
- Product reads/writes run through the RLS-scoped client.
- Raw secrets are not logged.
- Unit/integration tests cover both auth paths and one cross-workspace denial.
- A short developer doc explains how `CVS-100` routes should call the auth resolver.

### Suggested Linear Description

Implement the shared authentication foundation for Metrimap programmatic access. Programmatic MCP/API requests must authenticate via Clerk/OAuth or personal API key, resolve to a normalized auth context, enforce scopes and workspace/canvas access, and create an RLS-scoped Supabase client for downstream service calls. This issue should not build MCP tools or connector UX; it should give `CVS-100` through `CVS-104` a safe auth boundary.

### CVS-99A - Lock Programmatic Auth Model And RLS Strategy

Purpose:

Resolve the architectural ambiguity before implementation. The main decision is how OAuth/Clerk auth and personal API keys both become a safe, RLS-scoped Metrimap request context.

Problem:

The phrase "OAuth + key" hides important security decisions. If these decisions are left implicit, the MCP/API server can accidentally rely on service-role access, ambiguous workspace selection, or inconsistent permissions.

Scope:

- Decide whether v1 uses Clerk/OIDC directly, a Metrimap OAuth façade, or a staged Clerk-backed approach.
- Decide how personal API keys resolve to a user/workspace context without bypassing RLS.
- Define the normalized programmatic auth context.
- Define the v1 scope list.
- Define workspace selection behavior for multi-workspace users.
- Define which operations, if any, can use service-role access for narrow metadata lookup.
- Document rejected alternatives and non-goals.

Recommended v1 auth context:

```ts
type ProgrammaticAuthContext = {
  requestId: string;
  authMethod: "oauth" | "api_key";
  userId: string;
  clerkUserId: string;
  workspaceId?: string;
  scopes: string[];
  apiKeyId?: string;
  oauthConnectionId?: string;
};
```

Recommended v1 scopes:

- `mcp:read`
- `mcp:write`
- `canvas:read`
- `canvas:write`
- `metrics:write`
- `evidence:write`

Acceptance criteria:

- Decision doc exists in the repo or vault and is linked from the Linear issue.
- Decision explicitly answers OAuth issuer / Clerk delegation.
- Decision explicitly answers API key -> RLS context.
- Decision defines required claims and workspace selection behavior.
- Decision lists v1 scopes.
- Decision states that service-role product reads/writes are forbidden.
- `CVS-99B` and `CVS-99C` can proceed without re-opening architecture questions.

Non-goals:

- No implementation beyond lightweight spike code if needed.
- No MCP tools.
- No connector UX.
- No native source OAuth connectors.

### CVS-99B - Implement Programmatic Auth Resolver And Scoped Supabase Client

Purpose:

Build the reusable auth boundary used by MCP/API routes.

Problem:

MCP/API routes need one safe way to authenticate requests, enforce scopes, resolve user/workspace access, and create a Supabase client that obeys RLS. Without this shared resolver, each route may implement auth differently.

Scope:

- Implement a shared programmatic auth resolver.
- Accept OAuth/Clerk bearer tokens.
- Accept personal API keys.
- Hash API keys before lookup.
- Reject revoked, expired, malformed, or missing keys.
- Enforce scopes.
- Resolve user and workspace context.
- Reject ambiguous workspace selection.
- Confirm workspace/canvas access before product reads/writes.
- Create `createProgrammaticSupabaseClient(authContext)` or equivalent.
- Add safe errors for unauthenticated, forbidden, inaccessible, and unexpected failures.
- Add request metadata hooks for future audit logging.

Suggested functions:

```ts
resolveProgrammaticAuth(request): Promise<ProgrammaticAuthContext>
requireProgrammaticScope(context, scope): void
resolveProgrammaticWorkspace(context, target): Promise<WorkspaceContext>
createProgrammaticSupabaseClient(context): SupabaseClient
```

Acceptance criteria:

- OAuth/Clerk request resolves to a normalized auth context.
- API-key request resolves to the same normalized auth context.
- Both paths can create an RLS-scoped Supabase client.
- Missing auth returns `401`.
- Invalid/revoked/expired key returns `401`.
- Missing scope returns `403`.
- Inaccessible workspace/canvas returns `403` or safe `404`.
- Multi-workspace ambiguity is rejected until a target workspace is supplied.
- Raw API keys, access tokens, refresh tokens, service keys, and database URLs are never logged.
- Service-role usage, if any, is isolated to key lookup or narrow metadata and never used for normal product reads/writes.

Non-goals:

- No MCP server route scaffolding beyond a minimal protected test route if needed.
- No MCP tools.
- No connect/settings UX.
- No full audit UI.

Dependencies:

- `CVS-99A` must be complete.
- `CVS-89` must provide personal API key storage or a compatible contract.
- `CVS-98` should provide service-layer functions that can receive the auth context.

### CVS-99C - Add Auth Tests, Denial Cases, And Developer Usage Docs

Purpose:

Prove the auth foundation is safe enough for `CVS-100` through `CVS-104`.

Problem:

Programmatic auth is easy to get "working" while still failing important security cases. The implementation needs explicit tests and developer guidance before MCP tools are built on top.

Scope:

- Add unit/integration tests for OAuth/Clerk auth path.
- Add unit/integration tests for API-key auth path.
- Add cross-workspace denial test.
- Add missing-scope denial test.
- Add revoked/expired key denial test.
- Add unauthenticated request test.
- Add safe logging checks where feasible.
- Add developer docs showing how MCP/API routes must call the resolver.
- Add a short manual verification checklist.

Acceptance criteria:

- Tests prove both auth paths can reach a protected test route.
- Tests prove RLS or permission checks deny cross-workspace access.
- Tests prove insufficient scopes are rejected.
- Tests prove revoked/expired/invalid keys are rejected.
- Tests prove missing auth is rejected.
- Developer doc includes example route usage.
- Developer doc states non-negotiables: no service-role product reads/writes, no raw secret logging, always check workspace/canvas access.
- `CVS-100` has a clear integration contract for MCP server scaffolding.

Non-goals:

- No broad security review. That remains `CVS-104`.
- No MCP tool implementation.
- No production connect UX.

Suggested developer-doc example:

```ts
export async function POST(request: Request) {
  const auth = await resolveProgrammaticAuth(request);
  requireProgrammaticScope(auth, "canvas:write");

  const supabase = createProgrammaticSupabaseClient(auth);
  const workspace = await resolveProgrammaticWorkspace(auth, request);

  return createCanvasService({ supabase, auth, workspace }).handle(request);
}
```
