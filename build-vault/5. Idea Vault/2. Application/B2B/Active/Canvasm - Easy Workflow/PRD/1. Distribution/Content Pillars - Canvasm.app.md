# Content Pillars - Canvasm.app

This is the draft content architecture for the public Canvasm marketing site.

The goal is not to publish generic analytics content. The goal is to teach the worldview behind Canvasm: excellent growth, experimentation, and agentic work depends on the quality of the data foundation and the clarity of the metric map.

## Core Thesis

The next wave of growth teams will not win because they have more dashboards.

They will win because they know:

- what is instrumented,
- what the schema means,
- what sample size can and cannot prove,
- which metrics are unified,
- which relationships are believed,
- which beliefs have evidence,
- which experiments changed the system,
- and which agents can safely act on that model.

Canvasm exists because the future operating system for growth is not a dashboard. It is a living map of metrics, relationships, evidence, experiments, and decisions.

## Core Principle: Traceability Beats Artifact Creation

Claude Artifacts and similar AI artifact surfaces are useful because they let people create documents, apps, tools, code, diagrams, and interactive outputs quickly.

Canvasm should not compete by saying "we also generate artifacts." The stronger thesis is:

> The future problem is not only artifact creation. It is whether the artifact can be trusted, traced, explained, and acted on.

An AI-generated chart, notebook, strategy tree, workflow, or recommendation is only useful if the team can inspect where it came from:

- Which source data was used?
- Which instrumentation produced that data?
- Which consent, cookie, identity, and attribution assumptions shaped it?
- Which metric definition does it depend on?
- Which relationship does it claim is true?
- What evidence supports that relationship?
- What previous tests changed the team's conviction?
- Who owns the metric or workflow?
- What decision or action came from it?

This is the Canvasm difference:

| Surface | What it is good at | What Canvasm should add |
| --- | --- | --- |
| Claude Artifact | Creating and iterating standalone content, apps, tools, code, diagrams, and interactive outputs. | Provenance, source lineage, metric relationships, evidence, conviction, tests, owners, and business context. |
| Dashboard | Showing metric values. | Why the metric matters, what drives it, what evidence exists, and what to do next. |
| Notebook | Explaining analysis. | Linking the explanation to nodes, relationships, charts, sources, experiments, and decisions. |
| Project tracker | Tracking work. | Connecting work to metric movement, customer value, evidence, and workflow impact. |

Core traceability chain:

> Source -> Instrumentation -> Metric -> Relationship -> Evidence -> Conviction -> Artifact -> Decision -> Action -> Learning

Canvasm should make it hard for a team to say "the data is wrong" as a vague escape hatch. Sometimes the data will be wrong. But if every artifact is connected to source, relationship, evidence, conviction, and prior tests, the team can inspect where the weakness actually lives.

This becomes a core principle:

> Trust is not a vibe. Trust is traceability.

## The Big Narrative

Most teams ask: "What does the dashboard say?"

Better teams ask: "Can we trust the data?"

Excellent teams ask: "What does this metric mean, what causes it to move, what evidence supports that belief, and what should we do next?"

Canvasm should own that last question.

## The Content Ladder

The public content should move readers through this ladder:

1. Data foundation: instrumentation and schema.
2. Signal quality: size, sample rate, bias, latency, confidence.
3. Unification: warehouse, sources, semantic layer, and metric definitions.
4. Mapping: metric trees, relationships, dependencies, and operating models.
5. Evidence: notebooks, screenshots, charts, experiments, and decision records.
6. Action: experiments, strategy boards, alerts, and operating cadence.
7. Agents: AI and programmatic systems that can only be as useful as the map they act on.

The simple line:

> If the data foundation is weak, the agent is guessing. If the metric map is missing, the team is reacting. If the evidence is disconnected, the organization cannot learn.

## Use Case: Agentic Data Stack And External AI Workflow Mapping

Working title: Canvasm maps the workflow. Other tools run it.

Core argument:

Data-agent tools are collapsing the time it takes to generate pipelines, query warehouses, build forecasts, and create analysis. That is a real shift. But the bottleneck moves upstream: instrumentation, consent, identity, schema, source quality, definitions, governance, workflow design, and business strategy.

Canvasm should not own the data stack or run the workflow. Canvasm should own the map of why the stack or workflow exists, what metric it should move, what external system runs it, what evidence supports it, and whether it is working.

The hard truth:

> If the data flowing into BigQuery is from broken tracking, agents just automate bad answers faster.

What this pillar teaches:

- Instrumentation is still the foundation of every agentic system.
- Cookie management, consent, identity, and attribution are not boring implementation details. They decide what the business can truthfully know.
- Data engineering agents can help create pipelines, but humans still need to define the business model, events, metrics, owners, and action boundaries.
- The modern data stack needs a business architecture brief before teams commit to heavy implementation.
- Applied AI workflows should be mapped before they are automated in third-party tools.
- Action nodes should connect external agents, automations, scripts, and workflows back to metrics, evidence, guardrails, owners, and approvals.

Canvasm role:

- Visualize the value chain and identify what data, sources, or apps are needed to prove each link.
- Turn instrumentation and stack gaps into briefs for the user's chosen tools, teams, or agents.
- Let external agents use MCP/API to push created maps, values, evidence, and workflow status into Canvasm.
- Map AI orchestration across customer acquisition, support, retention, sales, and operations without running those workflows inside Canvasm.
- Connect every external workflow to the metric it should improve and the evidence that justifies it.

External workflow mapping fields:

- External runner: n8n, Zapier, Make, Intercom, HubSpot, Salesforce, WhatsApp provider, internal script, notebook, data agent, or custom service.
- Trigger: form submitted, WhatsApp message received, support ticket created, user crosses threshold, lead score changes.
- Source context: website, app, CRM, warehouse, ad platform, payment system, support tool, product analytics, social inbox.
- Action performed: send reply, classify, route, enrich, notify, create issue, update CRM, trigger email, escalate to human.
- Owner: person/team accountable for the external workflow.
- Target metric: conversion rate, response time, activation, retention, expansion, satisfaction, cost-to-serve.

## Product Model: Provenance, Evidence, And Conviction

Canvasm should make trust inspectable.

Important object model:

- Source: where the data, research, or artifact came from.
- Metric definition: what the number means and how it is calculated.
- Relationship: the claim that one node influences, composes, explains, or depends on another node.
- Evidence: the chart, screenshot, notebook, research note, transcript, experiment, or observation that supports a claim.
- Conviction: the team's current belief strength after reviewing evidence and previous tests.
- Artifact: a generated or human-created output, such as a notebook, chart, workflow, tree, report, or recommendation.
- Decision: what the team chose based on the artifact and evidence.
- Action: what changed in the product, workflow, campaign, pipeline, or customer operation.

Conviction should not be a static label. It should be a living state influenced by:

- Prior experiments.
- Metric movement.
- Sample size and signal quality.
- Source reliability.
- Relationship type.
- Conflicting evidence.
- Owner judgment.
- Time since last validation.

Possible conviction states:

- Unproven.
- Weak signal.
- Supported.
- Strongly supported.
- Conflicting.
- Stale.
- Invalidated.

This gives Canvasm a principled answer to "how is this different from an AI artifact?"

> Claude Artifacts help create the thing. Canvasm helps prove where the thing came from, what it depends on, and whether the team should trust it.

### Tip 2: Use MCP As The Handoff Layer

What the user does outside Canvasm:

- Architects or builds the data stack in their chosen harness: BigQuery, Supabase, Snowflake, dbt, Airbyte, RudderStack, n8n, notebook agents, data engineering agents, or internal scripts.

What Canvasm helps with:

- Defines the business map the stack should support.
- Exposes the expected metrics, relationships, sources, and evidence slots.
- Lets external agents/tools push created canvases, metric values, source summaries, architecture notes, and evidence back through MCP/API.

MCP skill idea:

- `architect_stack_for_canvasm`: reads a Canvasm map, identifies missing sources/events, proposes the user's external stack shape, and pushes a "stack architecture summary" back into Canvasm as evidence or source notes.
- `push_metric_values`: external tool computes or fetches metric values, then sends them to Canvasm.
- `push_evidence_note`: external agent creates a notebook-style explanation and links it to the relevant metric or relationship.
- `push_workflow_status`: external automation reports whether an Action node's external workflow is live, paused, failed, or retired.

Content angle:

- "Let your data agents build in the tools you trust. Let Canvasm hold the operating model."
