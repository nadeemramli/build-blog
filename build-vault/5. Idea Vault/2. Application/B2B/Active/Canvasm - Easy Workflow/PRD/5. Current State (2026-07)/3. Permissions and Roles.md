Not covered anywhere in this vault. The old PRDs assume single-owner canvases with no access tiers; this is the real, shipped permission model.

## Model

**Org-collaborative:** workspace (Clerk org) members always get full edit access. Roles only gate *guests* — people invited to a specific canvas who aren't members of the owning workspace (e.g. a personal-workspace canvas shared with an outside collaborator).

## The four roles

- **Viewer** — view-only banner on the canvas; can't drag, connect, or delete nodes; no comment composer.
- **Commenter** — comment-only banner; can post comments and @mention, still can't edit nodes/edges.
- **Editor** — full editing, no banner.
- **Admin** — behaves like Editor for canvas editing; member management stays owner-only regardless of role.

Role changes take effect immediately (guest sees the new behavior on reload, no re-invite needed). Enforcement is both client-side (banners/gating) and server-side (RLS rejects unauthorized writes even if the client is bypassed).

## Where it's set

People tab of the Collaborate panel (top-right of the canvas) — role dropdown per invited member, same options on the invite flow.

## Current status (as of 2026-07-04)

Shipped and covered by a manual two-user test plan (owner + guest in a different org). `npm run test:rls` exercises this at the RLS layer. Not yet built: any UI distinction between org-member roles (all members currently get full edit, no internal tiering within a workspace).
