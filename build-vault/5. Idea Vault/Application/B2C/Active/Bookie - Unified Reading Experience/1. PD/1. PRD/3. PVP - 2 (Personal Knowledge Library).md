# PVP - 2 — Bookie as a Personal Knowledge Library

Expansion of [[1. PVP]] (2026-07-12). The original PVP framed Bookie as reading + organization + Obsidian sync. This note raises the ceiling: **books become the first source of truth for knowledge in a specific domain/niche**, and Bookie becomes the user's mini-library where all quality knowledge lives, properly organized.

## The goal in one line

The user can shut down their brain, focus on fundamentals when needed, and go to Bookie for everything else — trusting that the knowledge is there, organized, and queryable.

## What this adds

1. **Knowledge groups.** Finished books (and other assets) can be grouped by domain/niche. A group is not just a shelf — it is a *corpus*:
   - Turn a group into a **conversational AI** (chat with "everything I've read about X").
   - Let an **agent scrape the whole library** for a specific problem the user is going through right now.
   - Grasp-of-knowledge tracking: what the user has actually read (and highlighted) within a domain, not just collected.

2. **Beyond books: articles and blogs as assets.** Blogs and articles are quality knowledge with a tendency to disappear from the web. Bookie should **scrape and save them as markdown** (with assets), so they are owned, permanent, readable, highlightable, and part of the same library/corpus as books. This is the Reader half of the reference products — but with permanence as the explicit goal, not just read-later.

3. **Reading remains the engine.** The intelligence layer only has value over what the user has actually ingested and read. Priority scoring, availability context, and manual progress from PVP-1 stay — they now also gate what enters a corpus ("done reading" is a meaningful state).

## Asset types (target)

| Asset | Ingestion | Canonical form |
|---|---|---|
| Books (EPUB/PDF) | upload / bulk import from local collection | original file + extracted text |
| Articles/blogs | web clipper / URL scrape | cleaned markdown + assets snapshot |
| Highlights/notes | created in reader | markdown (exports to Obsidian) |
| Future: newsletters, papers, videos-transcripts | same pipeline | markdown |

## Implications (see TRD - 2 for the architecture questions)

- The durable asset layer trends toward **files (markdown + originals) as source of truth**, with the database as index — this aligns with local-first, Obsidian, and agent access.
- Ingestion (parse, clean, dedupe, enrich) becomes a first-class subsystem, not an upload form.
- A knowledge index (text extraction → chunks → embeddings, local) sits beside the library index.
- The library should be exposed to agents (e.g. an MCP server over the library) — see [[Integrated MCP]] for earlier pointers (ebook-mcp, docsmcp).
