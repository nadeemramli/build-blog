Not covered anywhere in this vault (the closest thing, "Event Shocks" causal factors, is a passive annotation, not an active alert). This is a real, shipped monitoring layer on top of Metric Cards.

## What it does

Lets a metric card watch itself and notify someone when it crosses a threshold — closing the loop the old PRDs left open ("set alerts... for changes in Nodes, Updates in Threads, Data Source Issues" was listed as a to-do in the archived Features Pipeline notes, never designed).

## Rule types (set in card Settings → Alerts tab)

- **Value threshold** — e.g. value > 100.
- **Sharp change** — e.g. drops ≥ 10% period over period.
- **Outside a range** — min–max bounds.

Each rule can be enabled/disabled and deleted independently.

## Card states

- A muted **"Monitored"** badge once a card has at least one enabled rule.
- A red **"Alert"** badge once a rule has actually tripped.

## Delivery

- **Client-side evaluation** fires the moment a value changes (edit data, run a source, run an operator) — debounced so re-saving the same value doesn't re-fire.
- **Server-side sweep** via a scheduled `pg_cron` job (`metrimap-alert-sweep`, ~15 min interval) catches rules that already breach before the client ever re-evaluates them (e.g. a rule added retroactively to an already-bad value).
- Notifies both the **rule creator** and the **card owner** — surfaces in the notification bell inbox and the `/feed` Alerts tab (amber icon).

## Interaction with time-travel

Alert badges always reflect the **live** value, even while viewing a past period via value time-travel (see that doc) — browsing history doesn't retroactively change what's flagged as breaching now.
