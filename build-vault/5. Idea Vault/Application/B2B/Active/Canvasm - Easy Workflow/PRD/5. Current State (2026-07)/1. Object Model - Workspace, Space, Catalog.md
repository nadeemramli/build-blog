This replaces the flat "Canvas Project" model described in the older PRDs in this vault (`Early - PRD`, `Medium - PRD 1.0/2.0` — now archived). None of those describe this hierarchy; this doc exists to close that gap.

## Why this changed

The old model: `project` **is** the canvas — flat, 1:1, no folders, no org boundary. That stopped scaling once multiple people/canvases needed to share metrics and permissions. The repo's object-model decision (locked 2026-06-28, not to be re-litigated without explicitly superseding it) introduces nesting and a real semantic layer.

## The hierarchy

```
Workspace  ( = Clerk Organization )   — members, roles, invites, billing, data-source connections
   └─ Space / Folder                  — homepage folders + filter chips
        └─ Canvas  (what used to be called "project")
             └─ Node — placement of → Metric
```

- **Workspace = Clerk org.** Members/roles/invites are inherited from Clerk directly — this collapsed most of what would've been a custom "Organizations & Teams" build (the kind of thing sketched in `4. Product/1.a Homepage.md`) into just wiring up Clerk's org switcher.
- **Space** is a lightweight folder layer on the homepage — name + color, canvases can be moved into one, deleting a space falls its canvases back to "Uncategorized" (canvases are never deleted with the space).
- **Canvas** is the renamed "project" — unchanged conceptually, still the thing you open and edit.

## Two canvas use cases (this distinction matters everywhere below)

1. **Exploration** — build cards freely, no data, no source binding. Stays cheap and frictionless, lives per-canvas only.
2. **Operationalization** — wire a **Source Node → Metric Card**. This is the only thing that earns a place in the shared Metric Catalog.

## The Metric Catalog (promotion-based, not automatic)

The catalog is **earned**, not applied to every card. Three states:

1. **Exploratory / proxy** — no source binding, per-canvas only.
2. **Tracked candidate** — has a source binding + real data, auto-listed in a "Candidates" tab, but not yet shared.
3. **Catalogued Tracked Metric** — explicitly promoted via a "Catalog this metric" button. Now workspace-scoped and referenceable from any canvas in the workspace.

Referencing a catalogued metric elsewhere shares the **same definition and values** — it's a view onto one series, not a copy. Editing the catalog definition pushes down to every canvas referencing it (live on open canvases, on next load for others). A referenced card can be **detached** to fork an independent copy.

## Two-tier value storage

- Exploratory cards: inline `metric_cards.data` (no history cost).
- Tracked Metrics: a dedicated snapshot store (`{ period, value, snapshot_at, source, version }`), which is what powers freshness checks, stale/orphaned-binding states, and value time-travel (see the separate time-travel doc).

## What happens when a data source breaks

A bound node keeps its **last-known snapshot** and flips to a visible stale/orphaned state — never a silent zero or deletion.
