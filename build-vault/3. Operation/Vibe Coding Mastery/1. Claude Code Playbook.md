# Claude Code Playbook

Personal utilization playbook: how to *drive* Claude Code, ordered by adoption priority. Project-level standards (git baseline, CLAUDE.md/AGENTS.md, E2E, Linear wiring) are protocol, not preference — they live in [[6. Project Bootstrap Protocol]] and agents execute them at project start.

The maturity ladder this playbook climbs:

1. **Level 1 — Chat:** describe, paste, copy back. (Past this.)
2. **Level 2 — Pair:** agent edits files, I steer turn by turn. (Here today, July 2026.)
3. **Level 3 — Delegate:** agent owns a task end-to-end — plan, build, verify, self-review — I review outcomes, not keystrokes.
4. **Level 4 — Orchestrate:** parallel agents in worktrees, background routines, cross-model review, work arriving from Linear intake.

The bottleneck is almost never the model — it is whether repos, environment, and feedback loops let the model operate autonomously without breaking things.

## Tier 1 — Make every session start smarter (this week)

**Plan mode for anything non-trivial.** Start substantial tasks in plan mode (Shift+Tab): the agent researches and proposes before touching files. Reviewing a plan takes 2 minutes; unwinding a wrong implementation takes an hour. Rule of thumb: 3+ files or any schema/API change → plan first.

**Permission rules instead of prompt-fatigue.** Approving the same safe commands hundreds of times trains blind-clicking "yes" — the actual danger. Allowlist known-safe commands per repo (`/fewer-permission-prompts` scans transcripts and proposes the list automatically).

**Run `/init` on every not-yet-bootstrapped repo** and bring it up to the [[6. Project Bootstrap Protocol]] standard. The repos where agents already work best (twinagent, metric-mapping, main-blog) are exactly the ones with good context files.

## Tier 2 — Make quality automatic (this month)

**Hooks replace discipline.** A rule in a context file ("run analysis after every edit") is a *hope* the model may follow; a hook is a *guarantee* the harness enforces. Start with format-on-edit and type-check-after-edit on TS repos.

**`/code-review` before every merge.** A separate review pass with fresh context catches what the building session can't — the builder is anchored on its own choices. For big changes, ultra review (multi-agent, adversarial verification) is worth the cost. Pair with the cross-model ritual in [[2. Codex and Multi-Agent Strategy]].

**Verification is the agent's job, not mine.** The scripts exist (test:rls, lighthouse, check-links, screenshots). The habit change: end prompts with "…then run X and fix what fails" instead of running X myself afterward. An agent that closes its own verification loop is the difference between Level 2 and Level 3.

**MCP is already wired — use it in prompts.** Supabase, Linear, Vercel, and Canvasm MCP servers are connected. Prompts like:

- "Check the Supabase advisors and logs for this project, then fix the RLS warnings"
- "Pull the top 5 Linear issues in the current cycle and fix the two smallest"
- "Check the last Vercel deployment's build logs and fix the failure"

This closes the loop the Development Operating System describes — Linear intake → agent work → deploy verification — without me as the copy-paste middleman.

## Tier 3 — Parallelize and automate (this quarter)

**One agent per worktree.** Run independent agents on independent branches simultaneously: one on integrations, one on docs, one fixing bugs. Review and merge each as it lands. This is the honest version of what the old directory copies were trying to be.

**Subagents for research, main thread for decisions.** For "understand this codebase" or "find where X happens", have the session fan out subagents rather than filling the main context with file dumps. Keep the main thread's context for the actual build.

**Scheduled routines for maintenance.** Recurring work is agent work: nightly link-check on the marketing site, weekly dependency-update PRs, weekly Supabase-advisor review. `/schedule` creates cron-based cloud agents; the System Health Pipeline ([[2.a.iii System Health Pipeline]]) already defines what these routines should watch.

**Headless mode in CI.** `claude -p "prompt"` runs non-interactively — usable in GitHub Actions for failure auto-triage, changelog drafts, or first-pass PR review.

## Anti-patterns (the ones that actually cost me)

- **The 200-message session.** Context degrades; past ~50 exchanges on one problem, `/clear` and restate the goal in two sentences. Restating is cheap; a confused context is not.
- **Vague acceptance criteria.** "Make it better" produces plausible-looking drift. Every delegated task needs a checkable "done": tests pass, screenshot matches, endpoint returns X.
- **Skipping the diff review.** Level 3 means reviewing *outcomes* — it never means merging unread diffs. Review discipline is what makes delegation safe.
- **Letting the agent apologize instead of verify.** If it says "this should work now," the reply is "run it and show me."
