# Codex and Multi-Agent Strategy

I run both Claude Code and Codex. Used interchangeably they're redundant; used deliberately they're complementary. The value of a second model is **decorrelation** — two different models make *different* mistakes, so one catches what the other misses.

## Division of labor

| Situation | Tool | Why |
|---|---|---|
| Primary build sessions, long multi-step tasks | Claude Code | Deepest harness integration here: context files, hooks, MCP (Supabase/Linear/Vercel/Canvasm), skills, memory, worktree orchestration |
| Second-opinion review of a finished diff | Codex | Fresh context + different model = decorrelated reviewer. Reviews from the model that wrote the code share the writer's blind spots |
| Architecture decisions with real trade-offs | Both, independently | Ask both the same question *without showing either the other's answer first*. Agreement = confidence. Disagreement = the interesting part — that surface is usually where the actual risk lives |
| Cloud/background batch tasks (many small fixes in parallel) | Whichever has spare quota | Mechanical work doesn't need model loyalty; it needs throughput |
| Stuck: one model loops on the same failed approach | Switch models | A looping model rarely un-loops with more prompting. The other model breaks the frame — same trick as asking a colleague |

## The cross-review ritual

For any change that touches auth, RLS, payments, migrations, or public APIs:

1. Claude Code builds the change (plan mode → build → self-verify per [[1. Claude Code Playbook]]).
2. Codex reviews the diff cold: "Review this diff for correctness and security issues. Do not assume the approach is right." — no context about why choices were made, so it questions the approach itself.
3. Claude Code responds to the findings — *rebutting* the wrong ones, not blindly applying everything. A second model's review findings are hypotheses, not verdicts; auto-applying reviewer feedback just swaps one model's blind spots for the other's.
4. I arbitrate anything the two genuinely disagree on. This is the highest-value 5 minutes in the pipeline — disagreement between decorrelated models is the best bug-detector available.

The [[6. Project Bootstrap Protocol]] makes this possible: CLAUDE.md and AGENTS.md carry identical content, so both models see the same ground truth. Divergent context files are how two models end up building to different specs.

## Principles for adding more agents

More agents ≠ more progress. The scaling rules:

1. **Parallelism needs isolation.** Two agents in one working directory corrupt each other. One agent per worktree/branch, always.
2. **Decorrelation beats redundancy.** Three identical reviewers find the same bugs three times. Vary the model, the lens (security vs. correctness vs. performance), or the context given.
3. **Fan out reads, serialize writes.** Research, review, and analysis parallelize safely. Writes to the same surface serialize through branches and merges.
4. **The human owns the merge.** However many agents run, integration decisions stay with me. That's the control point that makes everything else safe — and where the Development Operating System's routing ([[2.c Agentic Triage Automation and Source Routing]]) connects: agents execute tasks, the system routes them, I own what ships.

## What to watch quarterly

Model capabilities shift fast enough that this division of labor should be re-audited roughly quarterly: which model is stronger on Rust (twinagent), which harness ships better background/cloud execution, whether a third tool earns a slot. The *structure* (primary builder + decorrelated reviewer + human merge) outlives any specific model ranking.
