# Task Sources and Intake Groups

This note classifies where product and development work comes from before it enters task intake.

The purpose is to avoid treating every signal as a random task. Each source has a theme, and each theme should be routed into the system differently.

Important correction: not every source should enter Linear immediately. Linear is strongest once something is or may soon become executable work. Some sources should first become evidence, product synthesis, or planning notes.

## The five intake groups

| Group | Theme | Sources |
|---|---|---|
| Strategy and Discovery | What should we build and why? | Planned product work, jobs to be done, competitive learning, market learning, experiments, product hypotheses |
| Product Evolution | What becomes necessary while building? | Feature completion, technical debt, UX polish, refactors, better workflows discovered during development |
| System Health | What is broken, risky, slow, or unsafe? | Runtime errors, reliability issues, performance issues, security issues, privacy issues, failed jobs, logs |
| Quality and Verification | What do we find when testing intentionally? | Manual QA docs, regression checks, edge cases, UI polish, visual bugs, acceptance test failures |
| User and Operations Signals | What do real use and operations reveal? | User feedback, analytics behavior, support needs, operational/admin needs, confusion points |

## Routing by source group

| Group | First home | Linear behavior | Automation level |
|---|---|---|---|
| Strategy and Discovery | Product vault planning | Create Linear work only after the idea becomes an initiative, experiment, research task, or scoped project | Low |
| Product Evolution | Build log, product vault, or Linear follow-up | Create Linear issues for concrete follow-ups; keep broader observations as build learning | Medium-low |
| System Health | Evidence store and Linear | Automatically group, score, and create or update Linear issues | High |
| Quality and Verification | QA doc or evidence store | Semi-automatically create or update Linear issues from structured findings | Medium-high |
| User and Operations Signals | Feedback evidence store and product vault | Cluster and synthesize first; create Linear work when the pattern is actionable | Medium |

See [[2.c Agentic Triage Automation and Source Routing]].

## Group 1 - Strategy and Discovery

This group creates direction.

Sources:

- planned product work
- jobs to be done
- user goals
- competitive or market learning
- experiments
- product hypotheses
- roadmap assumptions

Typical output:

- initiative
- project
- feature candidate
- experiment
- research task
- product principle

Obsidian is important here because the reasoning matters. Linear should track the work, but Obsidian should preserve why the work matters.

Most Strategy and Discovery signals should not start as Linear tasks. They should first be shaped in the product vault through planning, research, or synthesis. Linear starts when there is a concrete initiative, experiment, research task, or implementation package.

## Group 2 - Product Evolution

This group appears during building.

Sources:

- feature completion gaps
- technical debt discovered while implementing
- UX friction
- workflow improvements
- architecture simplification
- refactors that unblock future work

Typical output:

- feature improvement
- tech debt issue
- UX issue
- refactor task
- follow-up implementation task

This is the "we saw a better way while building" group. It should not always interrupt the current task. Most items should enter Linear triage unless they block the current work.

Some Product Evolution signals are better captured first in a build log or decision note. Create Linear work when the follow-up is concrete, actionable, and worth tracking separately.

## Group 3 - System Health

This group protects the product.

Sources:

- runtime exceptions
- failed jobs
- broken integrations
- slow pages or slow APIs
- build failures
- security risk
- privacy risk
- permission mistakes
- data inconsistency

Typical output:

- bug
- incident
- reliability task
- performance task
- security task
- privacy task

This group is the best candidate for automatic Linear creation because the evidence can be captured directly from the system: logs, screenshots, user action, browser/device info, version, affected route, and error fingerprint.

System Health is the best first source for agentic triage automation. It can be deduped by fingerprint, ranked by frequency and affected users, and synced into Linear as one parent issue per error group.

## Group 4 - Quality and Verification

This group comes from intentional checking.

Sources:

- manual QA docs
- acceptance criteria
- regression testing
- edge case testing
- visual review
- UI polish
- design consistency checks

Typical output:

- QA finding
- bug
- UI issue
- test coverage task
- acceptance criteria update
- manual test doc update

Design polish belongs here when the issue is visual correctness: spacing, alignment, mobile layout, broken states, contrast, overflow, empty state, or UI consistency.

## Group 5 - User and Operations Signals

This group comes from reality.

Sources:

- user feedback
- support requests
- analytics behavior
- usage drop-off
- repeated confusion
- admin workflow pain
- internal operations needs
- onboarding friction

Typical output:

- feedback item
- product insight
- UX issue
- admin feature
- support workflow improvement
- analytics investigation

This group often needs interpretation. A user request is not always the right solution. It is usually evidence of a job, friction, or expectation.

User and Operations signals should be clustered and interpreted before they become Linear tasks. A repeated user request may become a UX issue, research task, feature candidate, support workflow improvement, or product insight.

## Design polish split

Design polish can belong to two groups depending on the reason.

| Design issue | Group |
|---|---|
| User flow is awkward or confusing | Product Evolution |
| Visual layout is broken or inconsistent | Quality and Verification |
| User says the design feels unclear | User and Operations Signals |
| Design blocks trust or positioning | Strategy and Discovery |

## Routing rule

Every incoming item should get:

- source group
- source detail
- evidence
- affected product area
- suspected task type
- urgency
- confidence
- owner or next reviewer

Then route it to one of three places:

- Linear intake, when it is already likely to become executable work
- evidence store, when it is high-volume or needs grouping first
- product vault planning, when it needs synthesis before becoming work

After routing, executable work can enter [[2.b Task Normalization and Taxonomy]].
