# Obsidian Learning Loop

Obsidian should be the long-term memory and shared source of truth of the development system.

Linear is for work. Obsidian is for learning and product truth. The learning loop is what prevents humans and agents from repeatedly rediscovering the same lessons.

## Is Obsidian the best tool for this?

Obsidian is a strong fit when the goal is:

- long-term product memory
- connected thinking
- markdown files that remain portable
- decision logs
- build logs
- experiments and postmortems
- personal or team knowledge that compounds over time

Obsidian is not ideal as the main task tracker. Tasks should live in Linear. Obsidian should link to tasks when the thinking matters.

## Source of truth for agentic orchestration

The product vault should feed agentic orchestration.

Agents that work on different surfaces should share the same product understanding:

| Agentic surface | Example domain | What the agent needs from Obsidian |
|---|---|---|
| App | `use.canvasm.app` | product behavior, UX intent, known constraints, QA learning |
| Docs | `docs.canvasm.app` | feature behavior, terminology, user jobs, examples, limitations |
| Marketing | `canvasm.app` | positioning, audience, value proposition, proof, product narrative |
| Support or operations | internal tools | known issues, user confusion, workflows, playbooks |

Models, harnesses, prompts, skills, and workflow agents should sit on top of the product vault. They should not each build their own memory of the product from scratch.

When an agent produces useful learning, it should feed the vault:

- app agents feed implementation and behavior lessons
- docs agents feed clearer explanations and support gaps
- marketing agents feed positioning and conversion insights
- QA agents feed edge cases and regression patterns
- triage agents feed recurring issue patterns and prioritization evidence

## Product vault as Claude Code knowledge base

When Claude Code is used for implementation, the product-specific Obsidian folder should be granted as an additional directory. This lets the agent read product context and write useful learning without turning the repo into a product-memory folder.

Example product vault:

```text
5. Idea Vault/Application/B2B/Active/Canvasm - Easy Workflow
```

Claude Code should use the product vault for:

- context before shaping or implementing
- build logs after meaningful work
- decision updates
- QA learnings
- postmortems
- product insights
- architecture rationale
- docs and marketing context when those surfaces depend on product truth

The repo should only store product memory when it directly documents code behavior, architecture, or developer operations.

See [[3.b Claude Code Obsidian Knowledge Setup]].

## What should become an Obsidian note?

Create or update an Obsidian note when:

- a decision changes product direction
- a bug reveals a recurring system weakness
- a user signal changes your understanding of the product
- an experiment succeeds or fails
- a feature teaches something reusable
- an architecture choice needs future context
- an agent workflow produces a useful pattern
- a QA cycle reveals a category of edge cases
- docs, support, or marketing work reveals a clearer product explanation
- a surface becomes inconsistent with the product source of truth

Do not create a note for every tiny task.

## Note types

| Note type | Purpose |
|---|---|
| Operating system note | Reusable process or workflow |
| Product principle | Stable belief about how the product should work |
| Decision log | A specific decision, context, expectation, and later result |
| Experiment note | Hypothesis, test, result, learning |
| Postmortem | What happened, why, impact, prevention |
| Build log | Session-level progress and observations |
| Product insight | User, market, analytics, or support insight |
| Architecture rationale | Why the system is built a certain way |
| Playbook | Repeatable process for future work |

## Suggested folder placement

General reusable workflow:

```text
3. Operation/Development Operating System/
```

Product-specific application:

```text
5. Idea Vault/Application/<category>/<status>/<product-name>/
```

Each product-specific folder can include its own `CLAUDE.md` so Claude Code knows what to read first and where to write learning.

Raw build sessions:

```text
Product folder/Build Logs/
```

Decisions:

```text
Product folder/Decisions/
```

Experiments:

```text
Product folder/Experiments/
```

## Learning loop ritual

At the end of meaningful work, ask:

1. Did this change what we believe?
2. Did this reveal a recurring pattern?
3. Did this create a new principle?
4. Did this create a better process?
5. Would a future agent or builder benefit from this context?

If yes, capture it in Obsidian.

## Routing table: where each lesson type lands

The ritual questions above decide *whether* to capture. This table decides *where*:

| Lesson type | Destination | Example |
|---|---|---|
| Repo-specific fact or convention | That repo's `CLAUDE.md` / `AGENTS.md` | "RLS tests must run against local Supabase, never staging" |
| Cross-project workflow insight | `3. Operation` (DOS or Vibe Coding Mastery) | "Plan mode threshold: 3+ files or schema changes" |
| Product signal (user feedback, bug pattern, idea) | Product vault → Linear intake | "Users confuse driver nodes with metric nodes" |
| Recurring manual chore an agent could own | Skill, hook, or scheduled routine | Weekly Supabase-advisor review → scheduled agent |
| Environment/tooling incident | [[6. Project Bootstrap Protocol]] standing rules | The 738-duplicate-PATH incident |

The test for whether the loop works: **does the same mistake happen twice?** If an agent repeats an error a previous session already hit, the lesson wasn't written where the next session could see it.

## Session close: the agent does the routing

Before closing any substantial agent session (2 minutes):

> "Before we finish: what did we learn this session that the next session won't know? Update CLAUDE.md/AGENTS.md with repo facts, and give me anything vault-worthy."

The agent is better at summarizing the session than a human is at remembering it next week. Agent-side memory compounds automatically; the vault side only compounds if this prompt is asked.

## Weekly review ritual

Once per week:

1. Review completed Linear work.
2. Review bugs and QA findings.
3. Review user feedback and analytics.
4. Extract repeated patterns.
5. Update relevant Obsidian notes.
6. Create new Linear issues only for actionable work.
7. Update product principles or roadmap assumptions if needed.

## Link pattern

Use links in both directions:

- Linear issue links to Obsidian context.
- Obsidian note links to Linear issue IDs.
- PR links to Linear issue and important Obsidian notes.
- Postmortem links to bugs, PRs, deployments, and decisions.

## The closing loop

The loop is not closed when the task is done.

The loop is closed when:

- work is shipped
- evidence is reviewed
- lesson is captured
- future behavior changes because of the lesson

That is what turns development into compounding product knowledge.
