# Claude Code Obsidian Knowledge Setup

This note defines how Claude Code should use a product-specific Obsidian folder as the knowledge base and learning loop for a product surface repo.

The key idea: each repo remains the implementation truth for its surface, while the product vault becomes the long-term product memory that Claude Code can read and update when it has explicit filesystem access.

## Core pattern

For each product, connect three places:

| Layer | Location | Purpose |
|---|---|---|
| Work | Linear | Tasks, status, priority, owner, delivery state |
| Implementation | Surface repo | App code, docs content, marketing site, tests, migrations, or harnesses |
| Knowledge | Product Obsidian folder | Product memory, decisions, QA learning, build logs, postmortems, user insights |

Claude Code should be launched from the relevant repo, but granted access to the product Obsidian folder.

This pattern can apply to:

- app repo for `use.canvasm.app`
- docs repo or content system for `docs.canvasm.app`
- marketing repo for `canvasm.app`
- agent harness or eval repo that needs product truth

## Launch Claude Code with vault access

From PowerShell:

```powershell
$env:CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD = "1"
claude --add-dir "C:\Users\Nadeem\Desktop\Obsidian\build-blog\build-vault\5. Idea Vault\Application\B2B\Active\Canvasm - Easy Workflow"
```

From WSL:

```bash
CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1 claude --add-dir "/mnt/c/Users/Nadeem/Desktop/Obsidian/build-blog/build-vault/5. Idea Vault/Application/B2B/Active/Canvasm - Easy Workflow"
```

Inside an existing Claude Code session running from PowerShell:

```text
/add-dir "C:\Users\Nadeem\Desktop\Obsidian\build-blog\build-vault\5. Idea Vault\Application\B2B\Active\Canvasm - Easy Workflow"
```

Inside an existing Claude Code session running from WSL:

```text

```

If the session uses `/add-dir` after launch and the vault's `CLAUDE.md` was not loaded automatically, tell Claude Code to read the vault's `CLAUDE.md` before starting product work.

## Repo instruction file

Add a small `CLAUDE.md` or `.claude/CLAUDE.md` to the application repo.

Use it for stable project instructions, not long product memory:

```markdown
# Project Knowledge

This repo uses an external Obsidian product vault for product memory and learning.

When the product vault is available as an additional Claude Code directory:

- Read the product vault before shaping product work.
- Use the vault for product decisions, learning notes, build logs, QA learnings, docs context, marketing context, postmortems, and durable user insights.
- Do not create durable product-memory docs in the repo unless they are directly tied to code behavior or architecture.
- Link Linear issues, PRs, QA evidence, and release notes back to the relevant vault notes.

Product vault:

- Windows: `C:\Users\Nadeem\Desktop\Obsidian\build-blog\build-vault\5. Idea Vault\Application\B2B\Active\Canvasm - Easy Workflow`
- WSL: `/mnt/c/Users/Nadeem/Desktop/Obsidian/build-blog/build-vault/5. Idea Vault/Application/B2B/Active/Canvasm - Easy Workflow`
```

## Product vault instruction file

Add a `CLAUDE.md` inside the product Obsidian folder.

Example:

```markdown
# Canvasm Knowledge Vault

This folder is the product knowledge base for Canvasm.

Claude Code should use this vault to understand product context and preserve learning from implementation work.

## Read first

- Product overview or index note
- Current roadmap or active plan
- Recent build logs
- Open decisions
- Relevant QA docs
- Relevant user feedback or research
- Relevant docs, support, or marketing context

## Write learning here

- Build logs: `Build Logs/`
- Decisions: `Decisions/`
- QA learning: `QA/`
- User or operations insights: `Insights/`
- Postmortems: `Postmortems/`
- Architecture rationale: `Architecture/`
- Docs and support context: `Docs/`
- Marketing and positioning context: `Marketing/`
- Agent harness learnings: `Agentic Orchestration/`

## Rules

- Do not use this vault as the task board. Linear owns work state.
- Do not store secrets, tokens, raw private user data, or credentials.
- Link every implementation learning to the relevant Linear issue, PR, release, or QA evidence when available.
- Keep app, docs, and marketing terminology aligned with this vault.
- Ask before broad vault reorganizations, renames, or deletes.
- Prefer updating an existing relevant note over creating duplicates.
- Create new notes only when the learning will help future product, design, QA, or engineering decisions.
```

## Agent workflow

When Claude Code has product vault access, it should follow this loop.

### Start

- Read the Linear issue or user request.
- Read the repo context needed for implementation.
- Read the product vault index or `CLAUDE.md`.
- Read only the vault notes relevant to the task.
- Identify whether the work may create a learning note.

### During work

- Keep implementation changes in the repo.
- Keep tasks and status in Linear.
- Keep durable reasoning and product memory in the product vault.
- Keep app, docs, marketing, and harness work aligned to the same product source of truth.
- Avoid copying the same truth into multiple places.

### End

- Summarize code changes and tests.
- Link PR, Linear issue, and QA evidence.
- Update or create a product vault note only if the work produced reusable learning.
- Suggest follow-up Linear issues only when there is real actionable work.

## What belongs in the product vault

Use the product vault for:

- product principles
- decision logs
- experiment notes
- build logs
- QA findings that changed future testing
- postmortems
- user and operations insights
- architecture rationale
- recurring patterns
- playbooks for future agents
- docs and support explanations
- marketing positioning and terminology
- agent orchestration patterns

Do not use the product vault for:

- source code
- generated build output
- dependency lockfile notes
- temporary scratch notes that will not matter later
- private credentials or secrets
- task status that belongs in Linear

## Suggested folder structure

Inside a product vault:

```text
Product Overview.md
CLAUDE.md
Build Logs/
Decisions/
Experiments/
QA/
Insights/
Postmortems/
Architecture/
Docs/
Marketing/
Agentic Orchestration/
Playbooks/
```

The exact structure can vary by product. The important rule is that future agents can find the product context without scanning the whole vault.

## Relationship to Claude Code skills

The repo can still include Claude Code skills for repeatable workflows, such as:

```text
.claude/skills/system-health-intake/SKILL.md
```

Skills should describe how to perform a workflow. Harnesses and evals should test whether agents execute reliably. The product vault should store what the product has learned.

Example split:

| Content | Best place |
|---|---|
| "How to sync error reports to Linear" | Repo skill |
| "Canvasm export errors reveal this QA weakness" | Product vault learning note |
| "Run these commands before PR" | Repo `CLAUDE.md` |
| "Why we chose this workflow model" | Product vault decision note |
| "How docs should explain workflow automation" | Product vault docs note |
| "Which prompt harness checks docs consistency" | Agent harness repo or product vault orchestration note |

## Safety rules

- Grant Claude Code access only to the product vault needed for the current repo.
- Do not grant the whole Obsidian vault unless the task truly needs it.
- Ask before large-scale edits to the vault.
- Keep secrets out of Obsidian.
- Treat user feedback and error reports as sensitive evidence; summarize rather than dumping raw private data.
- Prefer links and summaries over duplicating full records.

## Definition of done for agent learning

The learning loop is complete when:

- the work item is updated in Linear
- the implementation is linked in the repo or PR
- verification evidence exists
- any durable learning is captured in the product vault
- the next agent can understand what changed and why without re-discovering the same context
