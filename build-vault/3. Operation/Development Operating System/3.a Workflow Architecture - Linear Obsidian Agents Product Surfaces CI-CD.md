# Workflow Architecture - Linear Obsidian Agents Product Surfaces CI-CD

This note explains how Linear, Obsidian, agentic work, the code repo, docs, marketing surfaces, and CI/CD should work together as one development system.

## Tool roles

| Tool | Role | It should answer |
|---|---|---|
| Linear | Work operating system | What are we doing, why now, who owns it, what state is it in? |
| Obsidian | Knowledge and learning system | What did we learn, why did we decide this, what should future work remember? |
| Agents | Execution and investigation layer | How can the work be planned, implemented, reviewed, tested, or summarized? |
| Code repo | Implementation truth | What actually changed? |
| Product surfaces | App, docs, marketing | Where product understanding is delivered to users |
| CI/CD | Verification and deployment gate | Does it build, pass, and deploy safely? |

## End-to-end workflow

1. Signal arrives from one of the source groups.
2. Source-specific routing decides whether it belongs first in Linear, an evidence store, or product-vault planning.
3. High-volume evidence is grouped and deduped before creating or updating Linear work.
4. Executable work enters Linear as Intake or Triage.
5. Intake is normalized using [[2.b Task Normalization and Taxonomy]].
6. If the item needs reasoning, context, or long-term memory, create or link an Obsidian note in the product vault.
7. The work is shaped into a clear issue with expected outcome and acceptance criteria.
8. A human or agent picks up the Linear issue.
9. Work happens in the code repo on a branch linked to the Linear issue.
10. Pull request links back to the issue and any relevant Obsidian context.
11. CI/CD runs automated checks and preview deployment.
12. The feature moves to In Review; a manual-test sub-issue is created in the Manual Test project, and a human verifies user-facing behavior (see Verification loop).
13. Release happens.
14. App, docs, and marketing surfaces are updated as needed.
15. Important learning is captured in the product Obsidian vault.
16. Future planning and agentic orchestration use the accumulated learning.

## Agentic triage layer

Linear should not receive one issue for every raw signal. For automated sources, especially System Health, use an agentic triage layer before or around Linear.

The triage layer should:

- group duplicate signals
- count occurrences and unique affected users or accounts
- infer product area
- suggest source group, task type, severity, priority, and confidence
- create one Linear issue per meaningful cluster
- update an existing Linear issue when new evidence arrives
- add rate-limited comments for important new occurrences
- keep raw evidence in the evidence store
- keep Linear focused on work that needs attention

Agentic triage can move items through Intake, Triage, Needs Evidence, or Escalation. It should not move complex work to Shaped or Ready without owner approval.

See [[2.c Agentic Triage Automation and Source Routing]].

## Obsidian setup

Obsidian should store product memory in the product-specific vault, not in the code repo.

The general operating-system notes live here:

```text
3. Operation/Development Operating System/
```

Product-specific learning should live in the relevant product folder, for example:

```text
5. Idea Vault/Application/B2B/Active/Canvasm - Easy Workflow/
```

The product vault should store:

- product principles
- decision logs
- experiment results
- postmortems
- build logs
- repeated user insights
- architecture rationale
- operating system notes
- product-specific adaptation notes

Do not use Obsidian as the main task board. Link to Linear when a note creates work.

## Docs policy — Obsidian vs repo

Documentation has two homes, and they must not overlap:

- **Obsidian (product vault) = product docs.** Product knowledge, PRD, feature explanations, decisions, learning, QA learnings, onboarding and positioning. This is where humans and agents read product *meaning*.
- **Repo `docs/` = infrastructure and structural technical docs only.** Things that must live beside the code and version with it: ADRs, environment/setup, auth integration, database/RLS, migrations, architecture and state-management guides. Nothing narrative or point-in-time.

Rules:

- Do not add feature narratives, changelogs, or point-in-time write-ups to the repo — write durable learning to the product vault instead.
- Do not duplicate the PRD or product principles in the repo; link to the vault.
- **Manual tests are Linear issues, never a compiled test-plan document.** A single all-in-one manual test doc is an anti-pattern — it goes stale and hides coverage. See the verification loop below.
- When a repo doc turns out to be product knowledge, migrate it to the vault and delete it from the repo.

## Product surface setup

The same product can have several user-facing surfaces:

| Surface | Example | Source of truth |
|---|---|---|
| App | `use.canvasm.app` | Product vault plus code repo |
| Docs | `docs.canvasm.app` | Product vault plus docs repo/content system |
| Marketing | `canvasm.app` | Product vault plus marketing site repo/content system |

The product vault should hold the shared product understanding. Individual repos or content systems should hold the implementation and presentation details for their surface.

This matters for agentic work: a docs-writing agent should not invent behavior from the code alone, and an app-building agent should not ignore positioning, terminology, support learnings, or docs promises. Both should read the product vault first when product meaning matters.

## Agentic orchestration setup

Agentic orchestration sits above the product vault.

It can include:

- models
- evals
- harnesses
- Claude Code skills
- MCP servers
- task-specific agents
- docs-writing agents
- marketing agents
- QA agents
- triage agents

These agents should use the product vault as shared context, then act on the appropriate surface. When their work reveals a durable product lesson, they should update the vault.

## Claude Code vault access

Claude Code should normally launch from the repo, with the relevant product Obsidian folder granted as an additional directory.

See [[3.b Claude Code Obsidian Knowledge Setup]] for the concrete setup.

Recommended pattern:

- repo `CLAUDE.md`: short instruction telling the agent which product vault to use
- product vault `CLAUDE.md`: product knowledge instructions and read/write locations
- Linear issue: current work state
- product vault notes: decisions, learning, postmortems, QA learnings, and product context

Do not grant Claude Code the entire Obsidian vault by default. Grant the specific product folder needed for the repo.

## Agentic work setup

Agents should start with:

- Linear issue
- expected outcome
- relevant Obsidian context
- product vault path if available
- affected surface: app, docs, marketing, or internal workflow
- repo link
- acceptance criteria
- test/QA expectations

Agents should end with:

- summary of changes
- linked PR or patch
- tests run
- risks or limitations
- follow-up issues if needed
- learning note update in the product vault if the work revealed something important
- surface updates completed or explicitly not needed

## Verification loop — the Manual Test project

Manual testing is routed by a dedicated **Manual Test** project + the `manual-test` label — **not** by a separate tester account. Any member (the owner) runs the tests; a second Linear user is unnecessary overhead.

The loop that makes "completed" mean "vetted":

1. An agent finishes a feature issue → opens a PR → moves the **feature issue to In Review** (never straight to Done).
2. The agent creates a **manual-test sub-issue** under the feature issue — title `Manual test: <feature>`, placed in the **Manual Test project**, labelled `manual-test`, and **assigned to the owner** (so it shows in their Todo) — containing the PR link and numbered test cases from the acceptance criteria (steps → expected → pass/fail).
3. A human opens the Manual Test project and runs the sub-issue. **Pass** → close it; the feature issue can move to Done (after merge/release). **Fail** → check off what passed, note the failure, and **spawn a follow-up bug sub-issue** for what broke; the builder iterates.
4. The loop ends only when the manual-test sub-issue passes and the change is merged/released.

This replaces both compiled manual-test documents and the earlier dedicated-tester account: every feature carries its own runnable test task in one project, coverage is visible, and failures continuously spawn follow-up sub-issues.

## Code repo setup

Repo conventions:

- branch name includes Linear issue ID
- PR title includes Linear issue ID
- PR description links Linear and Obsidian context
- tests and manual QA notes are included before merge
- release notes link important product changes

The repo should not become the place for product memory unless the memory is directly tied to code behavior or architecture.

## CI/CD setup

CI/CD should verify:

- build
- lint
- tests
- type checks
- migrations
- security checks where relevant
- preview deployment
- deployment status

CI/CD should feed back into Linear when something fails. Repeated CI failures should create a System Health item.

## Automation candidates

| Source | Automation |
|---|---|
| Runtime error | Create Linear bug with logs, route, user action, version, screenshot, fingerprint |
| User feedback | Create Linear feedback item with user quote, screen, account context, sentiment |
| Manual QA finding | Create Linear QA issue linked to the QA doc and test step |
| Analytics anomaly | Create investigation issue with metric, segment, timeframe |
| Feature reaches In Review | Create a `manual-test` sub-issue in the Manual Test project with test cases from the acceptance criteria |
| PR merged | Move Linear issue to QA or Released depending on deployment state |
| Deployment failed | Create or update System Health issue |
| Postmortem completed | Link Obsidian learning note back to related Linear issues |

See [[4.a System Health Task Intake Prompt]] for the implementation prompt for runtime ErrorBoundary reports flowing through Supabase into Linear.

## Critical rule

Do not duplicate the same truth across tools.

Linear says what work exists.
Obsidian says what was learned.
The repo says what changed.
CI/CD says whether it passed.
Product surfaces say what users can see and use.
Agents say what they did and what needs attention next.
