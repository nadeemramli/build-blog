# Task Normalization and Taxonomy

This note defines how raw signals become usable work.

A raw signal should not immediately become a build task. It should first be normalized so planned work, bugs, feedback, QA findings, experiments, and technical debt can be compared inside the same system.

Before normalization, the system should decide the signal's route. Some signals enter Linear intake. Some enter an evidence store for grouping. Some enter product-vault planning before they become work.

## Normalization fields

Every intake item should answer:

| Field | Purpose |
|---|---|
| Title | Clear short name |
| Source group | Strategy, Product Evolution, System Health, Quality, or User/Ops |
| Source detail | Feedback, error report, QA doc, roadmap, analytics, etc. |
| Product area | Which part of the app is affected |
| Task type | Bug, feature, UX issue, tech debt, research, etc. |
| Evidence | Screenshot, logs, user quote, metric, QA steps, repo link |
| Expected outcome | What should become true after the work |
| Severity | How bad the problem is |
| Priority | How soon it should be handled |
| Confidence | How sure we are that the issue and solution are understood |
| Effort | Rough size |
| Dependencies | What must be true first |
| Owner | Human, agent, or team responsible |
| Links | Obsidian note, Linear issue, PR, QA doc, release |

## Task taxonomy

| Type | Meaning | Typical source |
|---|---|---|
| Initiative | Large product direction or outcome | Strategy and Discovery |
| Feature | New user-facing capability | Strategy, Product Evolution, User Signals |
| Improvement | Better version of existing behavior | Product Evolution, Feedback |
| Bug | Something broken against expected behavior | System Health, QA |
| UX issue | Flow, comprehension, or interaction problem | Product Evolution, User Signals |
| UI issue | Visual layout, spacing, state, or consistency problem | Quality and Verification |
| Tech debt | Internal work that improves future changeability | Product Evolution |
| Performance | Speed, responsiveness, scale, or resource issue | System Health |
| Reliability | Failure rate, jobs, uptime, retry, recovery issue | System Health |
| Security | Auth, access, permissions, data protection issue | System Health |
| Privacy | Consent, data handling, exposure, retention issue | System Health |
| QA finding | Issue found through planned testing | Quality and Verification |
| Research | Work needed to reduce uncertainty | Strategy, Product Evolution |
| Experiment | Hypothesis with success/failure criteria | Strategy and Discovery |
| Chore | Maintenance, dependency, config, cleanup | Product Evolution |
| Release task | Migration, rollout, monitoring, rollback, announcement | Code Repo and CI/CD |
| Learning note | Captured insight, decision, postmortem, or principle | Obsidian Learning Loop |

## Linear status flow

Recommended statuses:

1. Intake
2. Triage
3. Shaped
4. Ready
5. In Progress
6. In Review
7. QA
8. Released
9. Learned or Archived

The last state matters. A task can be technically done but not yet learned from.

## Intake vs triage vs shaped vs ready

These four states are easy to blur, but they should mean different things.

| State | Meaning | Main question |
|---|---|---|
| Intake | Raw captured signal | What came in? |
| Triage | Sorted and judged signal | What is this, how serious is it, and what should happen next? |
| Shaped | Clear work package | What exactly should be built, fixed, tested, or learned? |
| Ready | Approved to be picked up | Is this sufficiently clear, prioritized, and unblocked? |

The workflow should protect the team from two failure modes:

- building raw signals before understanding them
- over-documenting every signal before knowing whether it matters

In practical terms, triage is the bundle of normalization, classification, deduplication, severity judgment, priority suggestion, and next-action selection. It is not just a Linear status.

## Deep dive - Triage

Triage is the sorting room.

The purpose of triage is not to solve the task. The purpose is to decide what the item is, how urgent it is, whether it is real, and what should happen next.

Triage should answer:

- Is this a duplicate of something we already know?
- Is this real, reproducible, or supported by enough evidence?
- Which source group did it come from?
- Which product area does it affect?
- What type of task is it likely to be?
- Is this urgent, important, both, or neither?
- Is this a build task, research task, QA task, support task, or learning note?
- Does it need more evidence before shaping?
- Who or what should own the next step?

### Triage inputs

Good triage needs enough context to avoid guessing:

- raw report or original signal
- source group
- user, account, environment, or segment if relevant
- screenshot, logs, metric, QA step, or quote
- affected page, feature, flow, or integration
- when it happened
- whether it happened once or repeatedly

### Triage outcomes

Every item in triage should leave with one of these outcomes:

| Outcome | Meaning |
|---|---|
| Shape | Worth turning into a clear work package |
| Needs evidence | Keep open, but collect logs, screenshots, reproduction steps, or user examples |
| Duplicate | Link to the parent issue and add the evidence there |
| Reject | Not useful, not real, not aligned, or too vague |
| Archive as learning | No task needed, but the insight should be captured in Obsidian |
| Immediate escalation | Security, privacy, data loss, outage, or severe user impact |
| Convert type | Example: user request becomes research, bug becomes UX issue, feedback becomes product insight |

### Triage checklist

- [ ] Source group identified
- [ ] Source detail captured
- [ ] Product area identified
- [ ] Evidence attached
- [ ] Duplicate check done
- [ ] Severity assigned
- [ ] Initial priority assigned
- [ ] Confidence assigned
- [ ] Next action chosen
- [ ] Owner or reviewer assigned

### Severity vs priority

Triage should separate severity from priority.

Severity asks: how bad is the problem?

Priority asks: how soon should we handle it?

Examples:

| Situation | Severity | Priority |
|---|---|---|
| Data exposure risk | Critical | Immediate |
| Checkout broken for many users | Critical | Immediate |
| Rare visual bug in admin page | Low | Low |
| Confusing onboarding copy causing drop-off | Medium | High |
| Internal refactor that unlocks next sprint | Low user severity | High priority |

This distinction matters because not every severe-looking issue is urgent, and not every high-priority item is a visible user bug.

### What triage should not do

Triage should not:

- design the full solution
- write a long PRD
- assign engineering before the item is understood
- merge several unrelated issues together
- turn every feedback item into a feature request
- let low-evidence items enter the build queue

Good triage creates clarity without pretending the work is already solved.

## Deep dive - Agentic triage

Agentic triage is the automated or semi-automated version of the sorting room.

It should help humans by doing the mechanical work:

- dedupe signals
- group related evidence
- count frequency
- count affected users or accounts when safe
- infer product area
- suggest source group and task type
- suggest severity, priority, and confidence
- link duplicates to a parent issue
- summarize evidence
- flag urgent risks

It should not replace judgment for:

- product strategy
- final roadmap priority
- feature scope
- acceptance criteria for complex work
- moving work to Ready
- rejecting ambiguous signals without a review path

For System Health, agentic triage can be high-confidence because fingerprints, stack traces, routes, and occurrence counts are structured. For Strategy and Discovery, agentic triage should mostly summarize and prepare planning context.

See [[2.c Agentic Triage Automation and Source Routing]].

## Deep dive - Shaped

Shaped means the item has become a clear work package.

An item is shaped when someone can read it and understand the problem, desired outcome, boundaries, acceptance criteria, and evidence without needing a long meeting.

Shaping should answer:

- What problem are we solving?
- Who is affected?
- What should become true after this work?
- What is in scope?
- What is out of scope?
- What evidence supports this?
- What is the smallest useful version?
- What are the acceptance criteria?
- What are the risks?
- What dependencies or blockers exist?
- Does this need product, design, engineering, QA, or security review?
- Should this create or update an Obsidian learning note?

### Shaped issue structure

A shaped issue should contain:

- problem statement
- expected outcome
- affected user or job
- affected product area
- evidence
- proposed approach or direction
- non-goals
- acceptance criteria
- QA notes
- dependencies
- risks
- links to Obsidian, repo, design, or previous issues

### Shaped vs ready

Shaped and ready are not the same.

Shaped means the work is understandable.

Ready means the work is approved, prioritized, and available to pick up.

An item can be shaped but not ready because:

- it is lower priority than current work
- it depends on another task
- design is not approved
- the team is waiting for user evidence
- it belongs in a later release
- it needs security or privacy review

### Example shaped issue

```markdown
# Export fails when proposal includes uploaded attachments

## Problem
Users cannot export proposals when the proposal contains uploaded PDF attachments.

## Expected outcome
Proposal export succeeds with uploaded attachments, or the user receives a clear recoverable error if an attachment cannot be included.

## Evidence
- 4 runtime error reports with same fingerprint
- 2 support messages from active users
- QA reproduced on Chrome and Edge

## Scope
- Fix export job for proposals with PDF attachments
- Add regression test for attachment export
- Improve error message if attachment processing fails

## Non-goals
- Redesign proposal export UI
- Add support for new attachment types
- Rework the full document pipeline

## Acceptance criteria
- [ ] Proposal with one PDF attachment exports successfully
- [ ] Proposal with multiple PDF attachments exports successfully
- [ ] Failed attachment processing returns a clear user-facing message
- [ ] Regression test covers attachment export
- [ ] Linear issue links PR, QA evidence, and release
```

## Definition of ready

An item is ready when it has:

- clear expected outcome
- known product area
- evidence or reasoning
- task type
- priority
- owner or next actor
- acceptance criteria
- no unknown blocker

## Definition of done

An item is done when:

- implementation is complete
- tests or QA evidence exist
- PR or deployment is linked
- Linear issue is updated
- any important learning is captured in Obsidian
- follow-up work is created only if it is real and necessary

## Prioritization model

A simple scoring model:

```text
Priority = user impact + business value + urgency + risk reduction + confidence - effort
```

Use this to support judgment, not replace it.

For automated System Health triage, use an evidence-based score:

```text
triage_score =
  severity
  + frequency
  + unique affected users/accounts
  + recency
  + core workflow impact
  + customer value impact
  + release regression signal
  + security/privacy risk
  - low confidence penalty
```

The score should rank the queue and suggest priority. It should not silently move complex work to Ready.

High priority usually means:

- blocks a core user job
- causes data loss or trust damage
- creates security or privacy risk
- affects many users
- blocks release
- unlocks important future work

Low priority usually means:

- cosmetic only
- unclear evidence
- low user impact
- speculative improvement
- high effort with weak confidence

## Deduplication rule

Many signals can point to the same underlying issue.

Examples:

- three user complaints
- one analytics drop-off
- one QA finding
- one runtime error

These should become one parent issue with multiple evidence links, not four disconnected issues.

## Relationship to Obsidian

Linear stores the normalized work item.

Obsidian stores the thinking when the item changes our understanding:

- why this matters
- what pattern it reveals
- what decision was made
- what principle should change
- what future agents should know

See [[3.c Obsidian Learning Loop]].

## Practical example - from raw signal to learning

Scenario: a user clicks `Export proposal` and receives a blank PDF.

### 1. Intake

The raw signal arrives from the app error reporter.

Captured automatically:

- route: `/proposals/123/export`
- action before error: clicked `Export proposal`
- browser: Chrome
- account type: paid workspace
- error fingerprint: `proposal-export-empty-pdf`
- screenshot: blank PDF preview
- app version: `2026.07.03`

Linear issue is created in `Intake`.

Initial title:

```text
Blank PDF after proposal export
```

### 2. Triage

The team reviews the item.

Triage findings:

- Source group: System Health
- Source detail: Runtime error
- Product area: Proposal export
- Task type: Bug
- Severity: High, because it blocks a core user workflow
- Priority: High, because it affects paid users and the export flow is core
- Confidence: Medium, because there is evidence but no confirmed root cause yet
- Duplicate check: 3 similar reports found

Triage outcome:

```text
Shape this into a bug fix. Merge the duplicate reports into this parent issue.
```

The issue moves from `Triage` to `Shaped` only after the problem is made clear.

### 3. Shaped

The issue is rewritten into a buildable package.

Problem:

```text
Proposal export sometimes returns a blank PDF when the proposal contains uploaded PDF attachments.
```

Expected outcome:

```text
Users can export proposals with attachments successfully. If export fails, the app shows a clear recoverable error instead of producing a blank PDF.
```

Scope:

- reproduce the blank export
- fix attachment handling in export job
- add regression test
- improve error handling
- update manual QA doc for proposal export

Non-goals:

- redesign export UI
- support new file types
- rebuild the document generation system

Acceptance criteria:

- [ ] Proposal without attachments exports correctly
- [ ] Proposal with one PDF attachment exports correctly
- [ ] Proposal with multiple PDF attachments exports correctly
- [ ] Export failure shows clear user-facing error
- [ ] Automated test covers attachment export
- [ ] Manual QA doc updated

Now the item is shaped.

### 4. Ready

The product owner or builder confirms:

- priority is high
- no design dependency
- no security review needed
- engineering can pick it up
- QA path is clear

The issue moves to `Ready`.

### 5. In progress

An agent or human developer picks up the issue.

They receive:

- Linear issue
- linked logs
- reproduction notes
- relevant code area
- manual QA expectations

Work happens in the repo on a branch linked to the Linear issue.

### 6. In review

The PR includes:

- code fix
- regression test
- error handling update
- note about risk
- link to Linear issue

CI/CD verifies:

- build
- tests
- lint/type checks

### 7. QA

Manual QA checks:

- export proposal without attachments
- export proposal with one PDF attachment
- export proposal with multiple attachments
- simulate attachment processing failure
- verify error message

QA evidence is linked back to Linear.

### 8. Released

The fix is deployed.

Linear moves to `Released`.

Monitoring checks:

- no new matching error fingerprints
- export success rate returns to normal
- no increase in PDF generation failures

### 9. Learned or archived

The team asks whether this changed understanding.

Learning:

```text
Proposal export needs fixture-based QA for every supported content block. The export pipeline can appear successful while producing invalid output, so success checks must validate output content, not only job completion.
```

Obsidian update:

- create or update a learning note about export pipeline QA
- update manual QA checklist for proposal export
- add principle: output-generating jobs need content validation, not only process validation

The loop is now closed.
