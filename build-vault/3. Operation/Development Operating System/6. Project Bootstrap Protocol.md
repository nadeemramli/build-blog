# Project Bootstrap Protocol

This note is a protocol, not an essay. It is written to be **executed by an agent**. When starting any new project (or adopting an existing directory), point the agent at this note and it should perform every step below, then report the checklist status.

**Invocation prompt:**

> Read "3. Operation/Development Operating System/6. Project Bootstrap Protocol.md" and bootstrap this project accordingly. Product vault folder: `<path under 5. Idea Vault, if this is a product>`. Stack: `<app | site | knowledge | other>`.

An agent that cannot complete a step must say so explicitly in the report — no silent skips.

## Step 1 — Version control baseline (non-negotiable)

1. If there is no `.git`: `git init && git add -A && git commit -m "baseline"`.
2. Create `.gitignore` before the baseline commit if missing (node_modules, .env*, dist, .next, __pycache__, .venv).
3. **Never leave a working directory unversioned** — the July 2026 audit found 6+ unversioned app copies; work in them was unrecoverable and invisible to agents.
4. Variants and experiments use branches or `git worktree add ../<name>-wt <branch>` — **never directory copies**. One agent per worktree is also the safe parallel-agent pattern.

## Step 2 — Agent context files

Create both `CLAUDE.md` and `AGENTS.md` with **identical content** (Claude Code reads the former, Codex reads the latter; divergent context files make two models build to different specs). Keep under ~100 lines, prune like code. Required sections:

```markdown
# <Project name>

## Commands
- dev / build / test / lint / type-check commands, exactly as run
- verification commands (E2E, RLS tests, screenshots, lighthouse)

## Architecture (5-10 lines max)
- where state lives, where API calls go, what each top-level folder means

## Conventions
- anything that differs from framework defaults

## Environment rules
- Secrets live in `.env.local` (git-ignored); a committed `.env.example`
  lists every required variable with placeholder values.
- Never write tokens/keys into this file, shell rc files, or vault notes.
  Machine-level secrets belong in `~/.config/secrets/env` (chmod 600).
- Never paste installer one-liners that append to shell rc files.
- Code stays on the Linux filesystem (~/workspace), never /mnt/c.

## Never do
- <generated files the agent must not touch, e.g. Prisma types, out/>

## Linear
- Project: <Linear project link>. New bugs/ideas found while working
  become Linear issues, not TODO comments.

## Product vault
- <path under 5. Idea Vault, if applicable> — read before shaping,
  write build logs / decisions / postmortems after meaningful work.
```

## Step 3 — Stack defaults

Unless the project brief says otherwise, use the house stacks (agents perform best on conventions already used across this workspace):

| Project type | Stack |
|---|---|
| Web app | Vite + React + TS + Supabase + Tailwind (+ Zustand; Prisma for typegen) |
| Docs / marketing site | Next.js + React + TS + Tailwind, deployed on Vercel |
| Knowledge site | Quartz |
| System tool / desktop | Rust (+ Tauri + Svelte if UI) |

## Step 4 — Verification scaffolding (E2E from day one)

A project is not bootstrapped until an agent can verify its own work without asking a human:

1. `lint`, `type-check`, `test` scripts wired and passing on the baseline.
2. **E2E**: Playwright installed with at least one smoke test (app boots, core page renders). Extend per feature — the E2E suite is the agent's definition of "working".
3. For Supabase apps: local stack via Supabase CLI, `test:rls` for row-level-security policies, `supabase gen types` scripted.
4. Web surfaces: `lighthouse` and `check-links` scripts (copy from metrimap-marketing).
5. Convention: agent prompts end with "…then run the verification scripts and fix what fails." Verification is the agent's job.

## Step 5 — Work intake and knowledge wiring

1. **Linear**: create the Linear project, link it in CLAUDE.md/AGENTS.md. Work arrives via the intake pipelines ([[2.a Task Sources and Intake Groups]], [[2.c Agentic Triage Automation and Source Routing]]).
2. **Product vault**: if this is a product, create its folder under `5. Idea Vault/...` with its own `CLAUDE.md` pointer, per [[3.b Claude Code Obsidian Knowledge Setup]] and [[3.c Obsidian Learning Loop]]. Grant it to Claude Code as an additional directory.
3. **Deployment**: Vercel project connected (web), preview deployments on PRs.

## Step 6 — Agent harness configuration

1. `.claude/settings.json`: permission allowlist for the project's known-safe commands (lint, test, build, typegen) so sessions don't train blind approval-clicking.
2. Hooks: format-on-edit and type-check-after-edit at minimum (guarantees beat instructions — a rule in a context file is a hope, a hook is a guarantee).
3. Plan-mode rule of thumb recorded in CLAUDE.md: 3+ files or any schema/API change → plan before building.

## Bootstrap report (agent returns this)

```text
[x] git baseline commit <hash>
[x] CLAUDE.md + AGENTS.md created (identical)
[x] stack: <chosen stack>
[x] lint/type-check/test passing
[x] E2E smoke test passing
[ ] Linear project linked — BLOCKED: <reason>
[x] product vault folder linked
[x] permissions + hooks configured
```

## Standing hygiene rules (context for why this protocol exists)

Absorbed from the July 2026 workspace audit:

- **Directory-cloning instead of branching** left 6+ unversioned app copies (metrimap family). Worktrees give the same "many live variants" workflow with one history. → Step 1.
- **Environment rot**: a mis-pasted installer line self-appended into `.bashrc` on every shell start (738 duplicate PATH entries), and a GitHub token sat in plaintext for a year. Agents inherit the environment; these rules now live in every AGENTS.md. → Step 2.
- **Archives**: retired projects become tarballs in `archives/` (build artifacts excluded, `.git` history included — a project isn't archived until committed). Quarterly: clear package caches, compact the WSL disk (`wsl --manage Ubuntu --set-sparse true` from PowerShell).

Personal utilization practices (how to *drive* agents, not what to build) live in `3. Operation/Vibe Coding Mastery` — [[1. Claude Code Playbook]] and [[2. Codex and Multi-Agent Strategy]].
