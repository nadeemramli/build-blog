# Implementation Roadmap

This note turns the Development Operating System into a practical rollout plan.

## Phase 0 - Decide the operating structure

Goal: make the system easy to use before adding automation.

Actions:

- create the `Development Operating System` folder in Obsidian
- define the five source groups
- define task taxonomy
- define Linear labels and statuses
- decide where product-specific notes should link back to the general system
- choose the product Obsidian vault path for each repo
- define how Claude Code gets access to that product vault
- decide which source groups enter Linear directly and which need evidence storage or product-vault planning first
- choose which notes are reusable and which are product-specific

Output:

- Obsidian hub note
- product vault path for the first repo
- repo `CLAUDE.md` or `.claude/CLAUDE.md` pointing to the product vault
- Linear taxonomy
- first set of templates

## Phase 1 - Manual workflow

Goal: prove the process manually before automating.

Actions:

- create Linear intake manually
- add source group and task type manually
- link Obsidian notes where reasoning matters
- require acceptance criteria before work begins
- require PR and QA evidence before closing issues
- capture weekly learnings in Obsidian
- launch Claude Code with access to the product Obsidian vault when agent work should use product context

Output:

- consistent task intake
- clearer prioritization
- fewer lost insights

## Phase 2 - Semi-automated workflow

Goal: automate capture while preserving triage quality.

Actions:

- one-click app error report to raw evidence and grouped Linear issue
- grouped System Health evidence store before Linear sync
- user feedback form to feedback evidence and product insight clustering
- manual QA finding template to QA evidence and grouped Linear issue
- PR links back to Linear
- deployment status updates Linear
- Obsidian note links added to Linear issues when available
- product vault notes updated by Claude Code for meaningful learning

Output:

- less manual capture
- better evidence on bugs and QA issues
- more complete work history

Runtime error automation should start with [[4.a System Health Task Intake Prompt]]: keep the browser-to-Supabase report flow, then use a server-side bridge to dedupe reports and create or update Linear System Health issues.

Triage automation should follow [[2.c Agentic Triage Automation and Source Routing]]: calculate grouping, frequency, affected users, severity, priority suggestions, and Linear updates automatically where the source evidence supports it.

## Phase 3 - Agent-assisted workflow

Goal: agents can execute work with context and return useful evidence.

Actions:

- agents read Linear issue and linked Obsidian context
- agents read the product vault through configured Claude Code additional-directory access
- agents create implementation plans
- agents update task progress
- agents open PRs
- agents summarize changes and tests
- agents update product vault learning notes when useful
- agents suggest follow-up tasks only when they belong in Linear

Output:

- faster execution
- better traceability
- clearer handoff between product thinking and implementation

## Phase 4 - Productized workflow

Goal: make the workflow part of the product experience.

Actions:

- in-app task intake system
- automatic classification of source groups
- automatic task taxonomy suggestions
- Linear integration
- Obsidian or markdown knowledge export
- repo and CI/CD integration
- agent handoff packets
- learning loop dashboards

Output:

- the development workflow becomes a product capability
- users can operate their own product learning loop

## Minimum viable version

Start with:

1. Linear intake statuses.
2. Source group labels.
3. Task type labels.
4. Obsidian hub and product vault learning notes.
5. Manual QA issue template.
6. Claude Code product-vault access for one repo.
7. One weekly learning review.

That is enough to make the system real.

## Practical rollout example

Start with one product area, not the whole company.

Example product area:

```text
Proposal export
```

### Week 1 - Manual intake and triage

Create Linear labels:

- `source:system-health`
- `source:qa`
- `source:user-ops`
- `type:bug`
- `type:ui`
- `type:ux`
- `type:tech-debt`
- `area:proposal-export`

Create statuses:

1. Intake
2. Triage
3. Shaped
4. Ready
5. In Progress
6. In Review
7. QA
8. Released
9. Learned or Archived

Run a simple daily triage:

- review new intake
- merge duplicates
- assign source group
- assign type
- decide whether to reject, gather evidence, shape, or escalate

### Week 2 - Shape work before building

For every high-value issue, require:

- problem statement
- expected outcome
- scope
- non-goals
- acceptance criteria
- QA notes
- evidence links

This is where raw signals become buildable work.

Example:

```text
Raw signal: User reports blank PDF.
Triage: System Health, bug, high severity, high priority.
Shaped: Export fails when proposal includes uploaded PDF attachments.
Ready: Approved for next engineering slot.
```

### Week 3 - Connect repo and QA

For each ready issue:

- branch includes Linear issue ID
- PR links back to Linear
- PR includes test evidence
- QA doc is updated if the user flow changes
- release is linked back to the issue

This makes the work traceable from signal to code to release.

### Week 4 - Capture learning

Review the shipped issues:

- Which problems repeated?
- Which source groups created the most work?
- Which issues had weak evidence?
- Which QA docs prevented regressions?
- Which product principles changed?

Capture only the useful learning in Obsidian.

Output:

- one updated product principle
- one updated QA checklist
- one learning note or postmortem if needed
- new Linear issues only for real follow-up work

## Later automation priority

Automate in this order:

1. Runtime error grouping and Linear issue updates.
2. Frequency, affected-user, severity, and priority suggestions for System Health.
3. Manual QA finding grouping and Linear issue updates.
4. User feedback clustering into product insights before Linear tasks.
5. PR and deployment status to Linear.
6. Agent handoff packets.
7. Product vault learning note suggestions.

The early automations should capture evidence, not make product decisions.
